# Project: Halloween Prop Controller

## Instruction for Codex

Read this document fully before making changes. 
Follow architecture and constraints strictly.

## Overview
This is a Raspberry Pi + Arduino-based Halloween prop controller.

- Raspberry Pi 5 runs a Flask web server
- Controls GPIO buttons (TRICK/TREAT)
- Plays audio via pygame
- Communicates with Arduino over serial (PROP_CTRL_V2 protocol)
- Arduino executes physical effects (relays, props)

## Current Status
- System boots automatically via systemd (halloween.service)
- WiFi configured and working
- Arduino connected at /dev/ttyACM0
- Audio system working
- Scene logic implemented and functional
- Password-protected web UI/API supported
- Serial reconnect and serial health reporting implemented
- Dual-TV video playback implemented on the Pi
- Ambient video is app-managed and always loops when idle
- Triggered video plays after the end of TRICK/TREAT show flows

## Architecture

## Serial Protocol (PROP_CTRL_V2)

Controller (Pi) sends commands:
- RUN:<scene>
- TOGGLE:<output>
- SYS:<command>

Arduino responds with:
- STATUS:<state>
- STATUS:RUNNING_SCENE:<scene>
- STATUS:RUNNING_SERVICE:<output>
- STATE:<output>:ON/OFF
- DONE:<command>
- ERROR:<reason>

Notes:
- Controller waits for DONE or ERROR to complete a command
- STATUS updates reflect current system state
- Outputs must always report state changes

## Scene Behavior

Scenes are executed in two layers:

1. Python (Pi):
   - Selects scene
   - Plays audio
   - Sends RUN:<scene> to Arduino
   - Manages sequencing (TRICK → DOOR, etc.)

2. Arduino:
   - Executes physical outputs
   - Controls timing of relays/effects
   - Reports completion via DONE:<scene>

Important:
- Python is responsible for orchestration
- Arduino is responsible for timing-critical execution


## Inputs

- GPIO buttons connected to Pi:
  - TRICK
  - TREAT

Behavior:
- Button press triggers run_show(mode)
- System ignores input if already running a scene
- STOP command cancels active show immediately

Current trigger debounce:
- HALLOWEEN_GPIO_BOUNCE_TIME defaults to 0.1 seconds

## System State Rules

- Only one scene can run at a time
- System must be IDLE before starting a new scene
- STOP must immediately cancel:
  - audio
  - active scene
  - Arduino outputs

- Arduino must always return to a safe OFF state after STOP

- Ambient TV video should resume after STOP, RESET, and ALL_OFF

## Audio System

- Audio is handled on the Pi using pygame
- Multiple channels are used:
  - MAIN (primary audio)
  - BACKGROUND (idle/fog sounds)
  - effect-specific channels

- Audio must:
  - not block scene execution
  - be stoppable via STOP command

Stereo split:
- Left channel = non-voice show audio / speaker feed
- Right channel = voice / ServoDMX AutoTalk feed

## Environment

- Runs on Raspberry Pi 5
- Python runs inside a virtual environment (venv)
- Service managed by systemd:
  - file: /etc/systemd/system/halloween.service

Key config:
- SERIAL_PORT = /dev/ttyACM0
- BAUD_RATE = 115200
- App tries SERIAL_PORT first, then detected /dev/ttyACM* and /dev/ttyUSB* ports

Arduino pin layout:
- HEAD_1 / Skinny = pin 4
- HEAD_2 / Hag = pin 5
- AIR_CANNON = pin 6
- AIR_TICKLER = pin 7
- DOOR = pin 8
- HORN / Ooga horn = pin 9
- CRACKLER = pin 10
- STROBE = pin 13
- FOG = pin 22
- TV_1 and TV_2 are ignored for now

Pi trigger wiring:
- TRICK dry contact = GPIO17 / physical pin 11 to GND
- TREAT dry contact = GPIO27 / physical pin 13 to GND
- Shared trigger ground can use physical pin 9 or 14
- Internal pull-ups are enabled in software; do not wire triggers to 3.3V or 5V

TV playback:
- The Pi drives two 1920x1080 TVs as one 3840x1080 X11 desktop
- Combined videos live in video/
- ambient.mp4 loops when idle
- triggered.mp4 plays once at the end of the show flow and then ambient resumes

Current show/video timing:
- TRICK: trick scene -> door sequence -> triggered video
- TREAT: door sequence -> trick scene -> triggered video
- Triggered video should not start early
- Ambient should not compete with the triggered video

Audio startup:
- App requests 100 percent system mixer volume on startup
- HALLOWEEN_SYSTEM_VOLUME overrides the requested percentage
- HALLOWEEN_SET_SYSTEM_VOLUME=0 disables startup volume changes

Web access:
- HALLOWEEN_ACCESS_PASSWORD enables password login for all UI/API routes
- HALLOWEEN_SECRET_KEY signs remembered-browser sessions
- HALLOWEEN_SESSION_DAYS controls remembered-browser lifetime

Video-related env vars:
- HALLOWEEN_VIDEO_DISABLED
- HALLOWEEN_VIDEO_PLAYER
- HALLOWEEN_VIDEO_DISPLAY
- HALLOWEEN_VIDEO_XAUTHORITY
- HALLOWEEN_VIDEO_AMBIENT_FILE
- HALLOWEEN_VIDEO_TRIGGERED_FILE

Stereo audio split:
- Left channel = non-voice show audio / speaker feed
- Right channel = voice / ServoDMX AutoTalk feed
- Do not mono-sum left and right channels together

## Constraints

- Do not move timing-critical logic from Arduino to Pi
- Do not block the main thread in Flask
- All serial communication must follow PROP_CTRL_V2 format
- System must remain responsive to STOP at all times
- Keep app.py and firmware.ino protocol expectations synchronized
- If display/video behavior changes, update README.md and this handoff
